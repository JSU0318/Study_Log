가독성, 유지보수성

```
이미 작성한 소스코드에서 구현된 일련의 행위들을 변경없이, 코드의 가독성과 유지보수성을 높이기 위해 내부구조를 변경하는 것
다시 말해 기능을 유지하되 읽기 좋고 지속적으로 관리하기 편하게 소스코드를 재작성하는 것이다.
```

사람, 협업

```
소프트웨어 개발을 위해 프로그래밍, 소스코드를 작성할 때 대부분 여러명의 사람과 함께 작업을 하게된다.
그리고 새운 사람이 내가 작성하는 프로젝트에 추가로 참여하게 되며, 인수인계가 되거나 불가능한 경우도 있음
```

리팩토링을 왜하는가?

```
소프트웨어 설계에서 질적향상을 위해 리펙토링을 함
(코드중복을 제거, 수정 용이성 향상)

소프트웨어 이해도를 향하 하기위해, 가독성 향상을 위해 함
본 프로젝트에 다른 사람, 다음 사람, 그리고 그 사람이 나일 수도있다.

버그를 찾는데 도움된다
프로그램 개발속도가 향상된다, 좋은 설계기반에선 개발 속도를 단축 할 가능성이 높아짐
```

리팩토링은 언제 하는가?

```
The Rule Of Three : 유사한 내용이 세번이상 반복 할 때, 리펙토링을 고려
똑같거나 비슷한 내용이 세 번이상 작성되어 있으면 무조건 하는 것이 아니라, 상황에 고려하여 리펙토링을 결정하는 것


새로운 기능을 추가할 때
지금 작성된 설계, 소스코드에서 새로운 기능을 추가하기 어려워 보이면, 리펙토링을 해야된다. 이러한 상황인 경우 유지보수성이 떨어지고, 가족성 역시 좋지 않을 수 있다.


코드리뷰를 할 때
협업하는 개발팀과 함께 코드리뷰는 질을 높일 수 있다. 그러나 인원이 너무 많은 경우네는 비효율적이므로 소수인원으로 진행하는 걸 권장한다
그러나 설계 리뷰에서는 많은 인원이 하여도 효과적이다.
(개발 프로세스에서 상위작업임을 고려해야함)
```
